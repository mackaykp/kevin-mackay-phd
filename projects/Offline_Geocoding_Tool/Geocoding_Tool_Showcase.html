Offline Geocoding Tool

Problem

Statistics Canada’s National Address Register (NAR) provides authoritative Canadian address and location data, but there was no lightweight, offline way to geocode large spreadsheets (e.g. school or facility lists) against it. The team needed a tool that could match input addresses to NAR, assign coordinates using a clear match hierarchy, and write results to CSV without depending on external APIs or GIS software.

Approach

I built an offline geocoding pipeline in R that uses a single DuckDB database of NAR (Address + Location) as the reference. The matcher follows a single hierarchy: exact full address → exact street (same geography) → relaxed (same province/city/postal, relaxed components) → fuzzy (narrow pool, composite score) → 6-digit postal centroid → FSA centroid → city centroid. Input is read from Excel or CSV; output is streamed to CSV with progress saved every 100 rows so runs can be resumed. The tool is designed to run with minimal setup: build the NAR DuckDB once, then point it at any file with Street, City, Province, and Postal Code columns.

More detail on design, scripts, and code is on the Project Overview and Scripts & Code pages. Below is the Final product: the code you run, the summary results, and maps for two real datasets.

Code to geocode

To run the tool on the SIF school data (or any similar file), you set the project root, source the geocoding script, and call geocode_excel() with the path to your input file. Output is written to output/ by default. No need to run this in the document—it’s shown here so you can see the workflow.

# Set project root (adjust path for your machine)
proj_root <- "C:/Users/macka/OneDrive/Documents/Cursor-Projects/Offline_Geocoding_Tool"
if (dir.exists(proj_root)) setwd(proj_root)

source("geocode_excel_file.R")

# Paths
xlsx_path <- file.path(proj_root, "data", "ungeocoded_files", "sif_data_table_2022_2023_en.xlsx")
nar_db    <- file.path(proj_root, "data", "nar.duckdb")

# Run geocoding (progress saved every 100 rows; re-run to resume)
result <- geocode_excel(
  xlsx_path,
  address_cols    = c("Street", "City", "Province", "Postal Code"),
  nar_duckdb_path = nar_db
)
# Output written to: output/sif_data_table_2022_2023_en_geocoded.csv

Generalization to other datasets

The SIF data use consistent column names and address formatting aligned with how Canadian schools are recorded. Other datasets often do not. Address fields may be combined (e.g. “Full address” in one column), use different conventions (e.g. “Municipality” vs “City”), or contain more variation in spelling and structure. The NAR reference uses Census Subdivision (CSD) names for geography; if the input uses different place names (e.g. township names or abbreviations), the narrow pool for fuzzy matching can miss valid candidates. So the same tool, without dataset-specific tuning, typically achieves lower match rates on other files.

The ODEF (Ontario Directory of Education Facilities) dataset is one such example: it includes a pre-existing “Full address” field and different column naming. Below are the same summary and map for ODEF. You can see more records falling back to postal or FSA centroids, and a lower share of exact or street-level matches. This illustrates both the tool’s reusability (it runs on any CSV/Excel with address-like columns) and the importance of input format and naming for best results.

SIF achieves a higher match rate and, where both true and geocoded locations exist, typically smaller distance errors. That is partly because the tool was developed and tuned on the SIF dataset: its column names (Street, City, Province, Postal Code) and address formatting match what the parser and match hierarchy expect. ODEF uses different fields (e.g. a combined full address, different column names) and often different place-name conventions. The same pipeline runs on both, but match quality and precision are better when the input aligns with the design assumptions.

Generalization and next steps. The tool shows that a single codebase can geocode multiple datasets, but generalizing to new or noisier data usually needs extra refinement. That can include: mapping input columns to the expected names, normalizing place names (e.g. township or abbreviation to Census Subdivision), or relaxing thresholds and pool sizes for fuzzy matching. Building the tool around one primary dataset (SIF) allowed a fast, working solution; extending it robustly to many dataset types would involve more parsing options, configuration, and possibly dataset-specific rules.
